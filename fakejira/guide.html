Settings & Customization

Click the gear icon on any board tab to open its settings. You can customize columns, priorities, label presets, and the board accent color. Use the "Defaults" button in the header to set project-wide defaults for new boards.

Data & Storage

All your data lives in your browser's localStorage. There is no server, no cloud sync, no account. This means your boards are fast and private, but the data only exists in this browser. Clearing your browser data or switching to a different browser means starting fresh.

If you have projects you care about, export backups using the Export button in the header. The exported JSON file is a complete snapshot you can re-import at any time.

Think of localStorage as a whiteboard, not a filing cabinet. Great for active work, but not a long-term archive. Save backups frequently.

How localStorage works
  • Key-value store built into every modern browser
  • Data is scoped to the domain -- other sites cannot read it
  • Typically capped at 5-10 MB per domain
  • Persists until you (or your browser) explicitly clear it
  • Not shared across browsers, devices, or profiles
  • Private/incognito windows use a separate, temporary store

Import / Export

Export your project as a JSON file from the header. When importing, you have three options: add tickets to the active board, add as a new board in the current project, or create an entirely new project.

The importer is happiest with one of two shapes: a bare JSON array of tickets for quick AI-generated backlogs, or a full project object that matches the app's export format. For AI workflows, target one of those two shapes exactly.

Older exported files are still migrated forward on import for backward compatibility, but newly generated AI files should target the current formats above instead of older schemas.

Multi-Project

Use the "New" dropdown to create additional projects. Each project gets its own URL (#project/0, #project/1, etc.) and is stored independently in localStorage. You can also duplicate an existing project or clear it from the same dropdown.

Generate Tickets with AI

Copy one of the prompts below and paste it into any LLM (ChatGPT, Claude, etc.) along with a description of your project. Save the JSON output as a .json file and import it using the Import button.

For best results, enable JSON mode / structured output if your model supports it. If it does not, these prompts still work because they define the exact output shape, valid enum values, and a concrete example to imitate.

File format audit: what the importer actually accepts
  • Simple format: a bare JSON array of ticket objects.
  • Full format: a project object with title, boards, activeBoardIndex, and defaults. Exported files also include version, and the importer accepts that too.
  • Backward compatibility: legacy versioned exports from versions 1, 2, and 3 are migrated to version 4 during import.
  • Not supported: an object like { "tickets": [...] } unless it is an older versioned export.
  • Default workflow columns: todo, in-progress, testing, done, canceled. todo is the default first column and canceled is the default last column.
  • Simple format minimum: the file must be a JSON array, and the first item must be an object with a title so import can start.
  • Simple format recommended fields: every ticket should include title, description, priority, status, labels, and optionally prompt.
  • Full format board fields: each board should ideally include id, title, color, tickets, columns, priorities, columnOrder, and labelPresets. Missing ids are auto-generated.
  • Full format ticket fields: each ticket should ideally include id, title, description, status, priority, labels, createdAt, and updatedAt. prompt is optional on import and defaults to an empty string, but include it whenever you want implementation-ready tickets.
  • Every ticket should include a real title. Missing or empty titles are imported as Untitled Ticket.
  • Ticket ids: any non-empty string works. The app generates ids with an FJ- prefix, but that prefix is a convention, not an import requirement.
  • Board ids: any non-empty string works. The app generates ids with a B- prefix, but that prefix is also just a convention.
  • status must match a board column id. Invalid statuses are remapped to the board's first column during normalization or add-to-board import.
  • priority must match a board priority id. Invalid priorities fall back to the board's first priority.
  • columnOrder should include every column id. Tickets missing from columnOrder are auto-added to the correct column based on status.
  • Missing ids, duplicate ids inside the same board, invalid timestamps, and missing prompts are auto-repaired on import, but you should still generate them correctly for the cleanest results.

Generates a bare ticket array. This is the easiest format for AI tools to produce reliably, and the importer accepts it directly.

Prompt 
You are generating an Agile This import file.

Return exactly one JSON array and nothing else.
- No markdown fences
- No commentary
- No comments
- No trailing commas

First, infer the project's main workstreams and likely implementation steps from the brief. Then produce a concrete backlog of non-overlapping tickets.

Output contract:
- Return 8-20 ticket objects.
- Each object must use exactly these keys:
  "title", "description", "priority", "status", "labels", "prompt"
- "title": short, specific task title written as actionable work
- "description": 1-3 sentences with scope, constraints, or acceptance criteria; use "" only when the title is already fully clear
- "priority": one of "low", "medium", "high", "critical"
- "status": one of the default column ids: "todo", "in-progress", "testing", "done", "canceled"
- "labels": array of 1-3 short lowercase strings
- "prompt": optional implementation prompt string; use "" when you do not have enough detail or when the brief does not call for execution-ready tickets

Quality rules:
- Prefer concrete engineering tasks over vague epics.
- Break large features into multiple tickets.
- Include setup, testing, documentation, polish, and deployment work when relevant.
- Default most new work to "todo" unless the brief clearly says something is already underway, in testing, done, or canceled.
- Treat "todo" as the normal starting column and "canceled" as a rare edge case.
- If the brief suggests coding or execution help, fill in "prompt" with clear implementation instructions for each ticket.
- Do not invent any extra keys.

Generates a full project file. Use this when you want boards, defaults, prompts, timestamps, and direct import as a new project.

Prompt 
You are generating a complete Agile This project import file.

Return exactly one JSON object and nothing else.
- No markdown fences
- No commentary
- No comments
- No trailing commas

Before writing JSON, infer the best project structure from the brief:
- Use 1-3 boards when separate workstreams or milestones make sense; otherwise use 1 board.
- Prefer realistic implementation tickets over vague themes.
- Include a ticket "prompt" whenever the brief calls for implementation-ready tickets, coding-assistant instructions, or execution detail. When you include it, write it as if giving work to a coding assistant: include the deliverable, important constraints, and what "done" looks like.

Required top-level shape:
{
  "title": "Project Name",
  "boards": [ ... ],
  "activeBoardIndex": 0,
  "defaults": {
    "columns": [ ... ],
    "priorities": [ ... ],
    "labelPresets": [ ... ]
  }
}

Required board shape:
- "id": unique non-empty string
- "title": string
- "color": hex color string
- "tickets": array of ticket objects
- "columns": array of column objects, each with "id", "name", "color"
- "priorities": array of priority objects, each with "id", "name", "color"
- "columnOrder": object whose keys are every column id and whose values are arrays of ticket ids
- "labelPresets": array of short strings
- Unless the brief strongly suggests a custom workflow, use the default columns "todo", "in-progress", "testing", "done", "canceled".
- In the default workflow, keep "todo" first and "canceled" last.

Required ticket shape:
- "id": unique non-empty string
- "title": string
- "description": string
- "status": must exactly match one of that board's column ids
- "priority": must exactly match one of that board's priority ids
- "labels": array of strings
- "createdAt": ISO 8601 timestamp
- "updatedAt": ISO 8601 timestamp
- "prompt": optional string; include it when the brief asks for execution-ready or AI-implementable tasks

Ticket shape example:
{
  "id": "TASK-1",
  "title": "Short task title",
  "description": "1-3 sentence task description",
  "prompt": "Implementation-ready instructions for an AI coding assistant",
  "status": "todo",
  "priority": "high",
  "labels": ["frontend", "auth"],
  "createdAt": "2026-03-08T10:00:00.000Z",
  "updatedAt": "2026-03-08T10:00:00.000Z"
}

Hard rules:
- Include every required top-level key.
- Include every required board key.
- Include every required ticket key.
- If you include "prompt" for one ticket, include it consistently for all tickets in that board.
- Make all IDs unique.
- Put every ticket id in exactly one matching columnOrder array based on its status.
- Keep defaults realistic for future boards. It is fine for a board to use the same columns/priorities as defaults.
- Unless the brief strongly suggests a custom workflow, use the default columns and default priorities.
- Do not add extra keys.
